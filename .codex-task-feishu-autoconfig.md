# ClawBox 飞书自动配置模块任务

现在不要继续做方向分析，按下面的任务拆解直接执行开发。原则：方案由我确定，Codex 负责实现。

当前目标：ClawBox 的快捷配置必须优先解决**飞书应用配置自动化**，`pairing approve` 这一步 MVP 允许用户手动操作，后续再图形化。

## 总目标
把当前“消息通道 -> 飞书快捷配置”改造成：**自动完成飞书应用/机器人配置 + 自动写入 OpenClaw 凭证**；完成后页面进入 **已配置，待配对** 状态。pairing 这一步本轮先做人类可操作引导，不强求自动批准。

## 任务拆解

### 1）重构快捷配置目标
- 不要再把快捷配置的主目标写成“扫码后直接接入完成”。
- 改成：**扫码后自动完成飞书应用配置**，完成后进入 **待配对**。
- 页面文案、状态文案、按钮文案都要对齐这个目标。

### 2）实现真实的快捷配置状态流（前后端）
- 将现有占位二维码/占位轮询，改造成清晰的状态机结构，至少包含：
  - `idle`
  - `pending_scan`
  - `scanned`
  - `authorized`
  - `provisioning`
  - `configured_pending_pairing`
  - `failed`
- 前端和后端接口都按这个状态流组织。
- 即使某些步骤暂时受限，也要把状态流、数据结构、错误信息组织正确。

### 3）飞书应用配置自动化：作为本轮主线
请优先在代码里落实/铺好这些能力（能真接就真接，接不完也要把接口和步骤完整铺出来，明确 TODO 与缺口）：
- 自动创建企业应用（或接入已有自动化创建流程）
- 自动启用 Bot 能力
- 自动导入/配置 OpenClaw 所需权限
- 自动配置事件订阅（长连接 + `im.message.receive_v1`）
- 自动推进到可用状态（发布/版本流程如果无法全自动，要明确缺口）
- 自动获取并保存 OpenClaw 所需配置字段，至少包括：
  - `appId`
  - `appSecret`
  - 以及你确认的其他必要字段（如果当前 websocket 模式不需要 `verificationToken`/`encryptKey`，就不要误导用户）

### 4）自动写入 OpenClaw / ClawBox 配置
- 快捷配置成功后，将凭证写入当前 ClawBox / OpenClaw 配置。
- 写完后要更新页面，使该通道显示为：**已配置，待配对**（不要直接显示已接入）。
- 已接入通道区域或状态区要能区分：
  - 已配置，待配对
  - 已接入

### 5）pairing 先做 MVP 引导，不强求自动
- 当前版本先不要把 `pairing approve` 作为必须自动化目标。
- 当配置完成后，页面明确提示下一步：
  - 去飞书里给机器人发送消息
  - 获取 pairing code
  - 运行命令：`openclaw pairing approve feishu CODE`
- UI 上要提供清晰的引导和可复制命令。
- 如果方便，可预留 pairing code 输入框或后续“一键批准”入口，但本轮不强求做完。

### 6）保留并整理手动配置
- 手动配置仍然保留。
- 手动配置完成后，状态也应该是**已配置，待配对**，而不是直接**已接入**。
- 这样手动配置和快捷配置在 pairing 前的状态一致。

### 7）状态与 UI 整理
- 消息通道页的状态语义统一：
  - 未配置
  - 配置中
  - 已配置，待配对
  - 已接入
  - 配置失败
- 已接入标签和卡片展示保持整洁；如果尚未 pairing 完成，就不要显示“已接入”。
- 之前已经调整过的：**已接入标签位置**、**编辑配置按钮命名与行为**，都保留。

### 8）验证要求
- 改完后自行做可用的语法检查/构建检查。
- 最后给我一份简短总结：
  - 改了哪些文件
  - 哪些能力已经真正实现
  - 哪些是接口/状态流已铺好但还需后续接通
  - 当前用户在快捷配置后具体还需要做什么

## 重要约束
- 不要再把重点放在 pairing 替代 `App ID/App Secret` 上，这个理解是错的。
- 重点是：**飞书应用配置自动化必须优先**；pairing 本轮先允许人工完成。
- 不要继续写泛泛分析，直接动手实现。

# ClawBox 飞书快捷配置修正任务（状态检测 + 真实二维码）

不要继续泛泛分析，直接执行修正开发。

## 目标
解决当前两个关键问题：
1. 已经配置并完成 pairing 的飞书通道，状态检测仍错误显示为“已配置，待配对”。
2. 点击快捷配置后显示的是占位二维码，不是真实可扫码内容；必须查清原因并着手修复，不能再用假二维码或“建设中”占位收尾。

## 任务拆解

### 1）修飞书通道状态检测
- 检查当前状态判断逻辑为什么只识别“已配置”而没识别“已配对 / 已接入”。
- 把当前正在使用、已 pairing 成功的飞书通道正确识别成“已接入”。
- 状态判断要明确区分：
  - 未配置
  - 已配置，待配对
  - 已接入
  - 配置失败
- 优先复用 OpenClaw 当前已有的 pairing 状态 / 命令 / 数据来源，不要自己瞎猜。

### 2）彻查“真实二维码为什么没生成”
- 找到当前快捷配置里占位二维码是在哪个文件、哪个接口、哪段逻辑里生成/返回的。
- 明确当前没有真实二维码的根因，至少说清是哪一类问题：
  - 没接真实二维码生成接口
  - 后端只返回占位数据
  - 有 token 但没编码成二维码
  - 缺环境变量 / 凭证 / 中间服务
  - 链路中断在某个接口
- 不允许只给模糊结论，必须定位到具体代码位置和缺口。

### 3）开始修“真实二维码生成链路”
- 目标不是再放一个假二维码。
- 能接真实链路就直接接；如果这轮仍接不完整，也至少要做到：
  - 后端返回真实的中间态数据 / token / 可编码内容，而不是占位假数据
  - 前端根据真实返回内容生成二维码
  - 如果链路缺后半段，也要展示真实错误/真实缺口，而不是展示看似可扫的假码
- 如果当前确实依赖某个环境变量 / app seed / 服务端凭证，必须把依赖写清楚，并在代码里用真实报错提示出来。

### 4）快捷配置 UI 要反映真实能力
- 不要再让用户看到“长得像能扫、其实没用”的二维码。
- 如果已拿到真实可扫码内容，就展示真实二维码。
- 如果还缺后半段链路，也要明确提示当前卡在哪一步、缺什么，不要演示化。

### 5）验证要求
- 改完后自行做语法检查/可用检查。
- 最后给我一份简短总结：
  - 改了哪些文件
  - 状态检测问题的根因和修复方式
  - 二维码问题的根因和当前修复到哪一步
  - 现在展示的是不是真实二维码
  - 如果还有剩余缺口，具体缺什么

## 重要约束
- 不允许再用假二维码糊弄。
- 不允许退回“快捷配置能力建设中”这种占位收尾。
- 重点是：查清根因、修状态检测、修真实二维码链路。
